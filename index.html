Lab Challenges

Problem 1 - Cybershujaa

Problem Statement: Perform OSINT reconnaissance on a target domain to enumerate public-facing assets, subdomains, employee emails, and exposed code repositories without violating scope.

Approach: Use passive information-gathering techniques: run subdomain enumeration, search certificate transparency logs, query public WHOIS/DNS, search social networks and Git hosting for leaked secrets, and produce an asset inventory. Respect scope and record evidence/notes for each finding.

Tools Used: whois, dig, crt.sh (Certificate Transparency), subfinder/amass (subdomain enumeration), theHarvester, Google dorking, LinkedIn/X searches, GitHub search, and a notes spreadsheet.

Screenshots: Include: subdomain enumeration output (amass/subfinder), crt.sh certificate entries showing subdomain, sample LinkedIn/OSINT result, and a saved CSV of discovered hosts.

Key Lessons Learned: Passive OSINT yields a surprising number of exposed assets; certificate transparency and Git search are high-value. Always validate ownership and stay strictly within an authorized scope. Document sources and timestamps for reproducible reporting.

Problem 2 - Cybershujaa

Problem Statement: Scan and enumerate a lab network to identify live hosts, open services and versions, then prioritize likely vulnerable targets for further testing.

Approach: Perform ICMP/TCP discovery to find live hosts, run port scans on discovered hosts, run service/version detection, and correlate results with known CVEs. Triage targets by criticality (exposed RDP/SMB, outdated web servers, public-facing databases).

Tools Used: nmap (host discovery, -sS, -sV, -O), masscan (fast discovery), nmap scripts (NSE) for additional checks, and Vulners/OSVDB lookup for CVE mapping. Use a local notes file to track CVE IDs and remediation suggestions.

Screenshots: nmap summary of open ports and service versions, NSE script output flagging weak ciphers or outdated software, and a short prioritized target table exported to CSV.

Key Lessons Learned: Accurate versioning speeds exploitation planning; noisy scans (masscan) should be coordinated with stakeholders. Automating CVE lookups saves time but always manually verify exploitability and impact.

Problem 3 - Cybershujaa

Problem Statement: Identify and exploit a SQL injection vulnerability in a deliberately vulnerable lab web application, extract limited data, and prepare a concise remediation report.

Approach: Start with manual input testing and error observation, use proxy (Burp) to intercept and manipulate requests, fingerprint input points, confirm injection using time-based and error-based techniques, extract limited columns with safe UNION/boolean queries, and stop after proof-of-concept data to remain within lab rules.

Tools Used: Burp Suite Community (interception, repeater), sqlmap (automated confirmation & controlled extraction with `--risk`/`--level` safe settings), browser DevTools, and a local VM for testing payloads.

Screenshots: Burp request showing the vulnerable parameter, sqlmap output confirming injection and rows extracted, and the PoC request/response demonstrating a successful boolean/time-based test.

Key Lessons Learned: Manual testing reveals context that automation misses (e.g., input filtering). Limit extraction to proof-of-concept and record exact payloads; suggest parameterized queries, ORM protections, and input validation as remediation.
